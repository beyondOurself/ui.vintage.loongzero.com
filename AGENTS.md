# AGENTS

> 用于约定本项目长期适用的**开发规范、流程约束、协作规则**等基础原则。
> 与 RESEARCH.md（方案/结论沉淀）、TODO.md（执行性业务进度）联动，为原型迭代、页面实现和团队协作提供统一工作基线。
> 仅记录项目长期约束和可复用标准，**不存放临时方案、日常进度、讨论记录**。


## 项目概述
- 古着商场APP —— 高保真UI原型，采用 Vue 3 + UnoCSS + 原子类设计，搭建 vintage 二手潮流电商移动端。聚焦：极简体验、实用交互、可多端发布，统一页面风格与字段语义，对接后端商品、订单等主业务流程。

## 主题与 UI 规范（强制）
- 每个主题目录 `themes/<主题id>/` 下均有对应的 **`DESIGN.md`**，为该主题的 UI 设计系统单一事实来源（色板、字体、字号层级、间距、圆角、 elevation、组件形态、Do/Don't 等）。
- **生成或修改该主题下任意 HTML/CSS 时，须先阅读并严格遵守该主题自己的 `DESIGN.md`**，禁止套用其他主题的视觉习惯（如把暖色大地的阴影/色值抄进 MarketNest 等）。
- 实现要点：内联原子类与 `styles.css` 中的色值、圆角、边框/阴影策略须与当前主题 `DESIGN.md` 一致；缺省或自创配色、与规范冲突的交互样式视为不合规。
- 若某主题尚未提供 `DESIGN.md`，须先补齐规范文档再改该主题页面，不得无规范盲改。

## 开发规范
- 默认最小改动
- 优先修改指定文件
- 不改无关文件

## 环境约束（Windows）
- OS: Windows
- Shell: PowerShell
- 命令兼容：优先 PowerShell 语法

## 代码风格
- 代码应高效、规范，结构清晰、易于理解和维护。
- 命名简洁有意义，采用驼峰（camelCase）或短横线（kebab-case）风格，保持一致。
- 缩进统一为 2 空格，不使用制表符（Tab）。
- HTML/CSS/JS 按功能和结构合理分组，避免重复和大段无效注释。
- 注释应简明扼要，仅在必要处解释复杂逻辑。
- 保持文件结构简洁，目录命名清晰，避免多级嵌套。
- 推荐使用 **UnoCSS（Wind3，Tailwind v3 兼容 utility）** 原子类为主，定制样式集中于各主题 `styles.css`；改类名后执行 `npm run uno` 更新 `uno.css`。
- 生成的图标风格要统一，保持全站视觉风格一致。
- `icons/` 内 SVG 须兼容 [iconfont](https://www.iconfont.cn/) 上传：单色、`fill="#000000"`（不用 `currentColor`）、结构为 `<svg><path/></svg>`、无多余包裹 `<g>`、UTF-8 无 BOM；better-icons 等下载后按此整理再入库。
- 代码提交前确保通过本地检查，避免低级语法错误和拼写问题。

## 提交规范
- 采用「类型(范围): 内容摘要」方式（[Conventional Commits](https://www.conventionalcommits.org/zh-hans/v1.0.0/)）。
- 中文为主，内容摘要清晰说明变更点，如“feat(index): 新增底部 tab 导航”，“fix(cart): 修复结算栏遮挡问题”。
- 类型常用：`feat`（新功能）、`fix`（修复）、`docs`（文档）、`style`（样式/无逻辑）、`refactor`（重构）、`perf`（性能）、`test`（测试）、`chore`（构建/工具/杂项）。
- 范围用小括号表明受影响页面、组件或模块，必要时可细化（如：`feat(detail): 优化图片预览交互`）。
- 单次提交聚焦单一改动，避免无关内容混杂；能拆则拆，保证追溯。
- 重大功能或视觉方案变更须在 commit 描述补充“变更影响”说明，或在 PR/合并说明文档中补充详细内容。
- 严禁无意义提交（如 `fix: .`、`test`、`update`、`修改文件` 等笼统描述）。

- 在不同设备开始开发前，先读取 `TODO.md` 了解最近业务进度
- 以 `TODO.md` 作为业务任务状态单一事实来源
